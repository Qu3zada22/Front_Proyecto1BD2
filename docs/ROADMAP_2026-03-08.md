## Prioridad ALTA — Corregir antes de entregar

### [FIX-01] Eliminar índice único de resenas (BUG-01)

- **Archivo**: `apps/api/src/resenas/schemas/resena.schema.ts`
- **Cambio**: Eliminar el bloque `ResenaSchema.index({ usuario_id, restaurante_id }, { unique: true, sparse: true, ... })`
- **Razón**: 80 pares duplicados en DB — el índice nunca se aplica y causa warnings de startup
- **Impacto tests**: Actualizar `resenas.service.spec.ts` si lo referencia

### [FIX-02] Corregir text index de restaurantes (BUG-02)

- **Archivo**: `apps/api/src/restaurantes/schemas/restaurante.schema.ts`
- **Cambio**: Eliminar `weights: { nombre: 10, descripcion: 5 }` del índice de texto
  (alinear con lo que `indexes.js` crea, que ya existe en la DB)
- **Razón**: MongoDB rechaza crear índice de texto con opciones distintas al existente

### [FIX-03] Añadir `fecha_creacion` a MenuItem schema (BUG-03)

- **Archivo**: `apps/api/src/menu-items/schemas/menu-item.schema.ts`
- **Cambio**: Añadir `@Prop({ default: () => new Date() }) fecha_creacion: Date;`
- **Razón**: DB tiene `fecha_creacion` en los 72 items; nuevos items creados vía API no la tendrían

### [FIX-04] Corregir ObjectId cast en `removeByRestaurant` (BUG-04)

- **Archivo**: `apps/api/src/menu-items/menu-items.service.ts:70`
- **Cambio**: `{ restaurante_id: new Types.ObjectId(restauranteId) }`
- **Razón**: String vs ObjectId → 0 eliminaciones silenciosas
- **Test**: Añadir test que verifique que se pasa ObjectId al deleteMany

### [FIX-05] Añadir `en_proceso` al enum Swagger de ordenes (BUG-05)

- **Archivo**: `apps/api/src/ordenes/ordenes.controller.ts:26`
- **Cambio**: Añadir `'en_proceso'` al array de enum del `@ApiQuery`
- **Razón**: 4 991 órdenes con este estado no aparecen en la documentación Swagger

## Prioridad MEDIA — Mejoras de consistencia

### [CLEAN-06] Eliminar `timestamps: true` de MenuItem schema

- **Archivo**: `apps/api/src/menu-items/schemas/menu-item.schema.ts`
- **Cambio**: Cambiar `@Schema({ timestamps: true, ... })` → `@Schema({ timestamps: false, ... })`
- **Razón**: DB usa `fecha_creacion` (explícito), no `createdAt`/`updatedAt`. Con FIX-03 aplicado, `timestamps: true` solo añade campos innecesarios.

### [CLEAN-07] Añadir índice `veces_ordenado_desc` al schema de MenuItem

- **Archivo**: `apps/api/src/menu-items/schemas/menu-item.schema.ts`
- **Cambio**: Añadir `MenuItemSchema.index({ veces_ordenado: -1 }, { name: 'idx_menuitems_veces_ordenado' })`
- **Razón**: El índice existe en DB (creado por `indexes.js`) pero no está declarado en el schema

### [CLEAN-08] Alinear `indexes.js` con los schemas de Mongoose

- **Archivo**: `apps/database/operations/indexes.js`
- **Cambio**:
  - Añadir weights al text index de restaurantes: `{ nombre: 10, descripcion: 5 }`
  - O remover del schema (FIX-02 ya lo hace)
  - Añadir `propietario_id+activo` compound a restaurante schema (o eliminarlo de indexes.js)

## Prioridad BAJA — Deuda técnica

### [DEBT-09] Considerar eliminar `timestamps: true` en `ordenes` y `restaurantes`

- Ambos tienen `timestamps: true` + campo `fecha_creacion` explícito → docs nuevos tendrán ambos campos
- Propuesta: usar `timestamps: false` y depender solo de `fecha_creacion`
- **Riesgo**: Cambio mayor, requiere revisión de todos los queries

### [DEBT-10] Renombrar parámetro `cliente_id` → `usuario_id` en `ordenes.service.findAll`

- **Archivo**: `apps/api/src/ordenes/ordenes.service.ts:51`
- **Cambio**: `query.cliente_id` → `query.usuario_id` para consistencia con DB
- **Razón**: El campo en DB es `usuario_id`, el parámetro de query es `cliente_id`
- **Impacto**: Actualizar el controller y los tests
